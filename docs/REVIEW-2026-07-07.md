## 🟡 Medium severity

### 1. Per-chunk `files` upsert — N identical writes where 1 suffices
**`src/sift/adapters/store/libsql.py:331` · efficiency**

`_upsert_job` runs `_INSERT_FILE` *inside* the per-chunk loop (lines 331–334). But `upsert()` is called once per document with that document's whole chunk list (`ingest.py:118–125`), so every chunk shares the same `source_hash`/`source_path`/`modified_at`. A 200-chunk document therefore fires **199 redundant, byte-identical `files` upserts** (each an `ON CONFLICT DO UPDATE` re-writing the same row) — real write amplification against remote Turso, where each `execute` is a round-trip. The per-chunk `_INSERT_CHUNK` is correct; only the file upsert is redundant.

**Fix:** Hoist the file upsert out of the loop — dedupe by `source_hash` in one pass and execute `_INSERT_FILE` once per unique hash (or `executemany`). Collapses N file writes → 1 per document, and stays correct for hypothetical multi-doc batches.

### 2. Watch mode re-hashes the entire tree on every change
**`agent/sync.py:334` · redundant-recomputation**

In `--watch`/desktop mode, every debounced FS event (~1.5s) triggers `sync()` → `collect_roots()`, which calls `_sha256_file()` — a **full file read + SHA-256** — for *every* matched file. There's no `(path, mtime, size) → hash` cache anywhere; the long-lived process keeps only a set of paths. So editing one small file in a large watched corpus re-reads and re-hashes the **whole corpus** from disk, on every change, for the life of the process.

**Fix:** Keep an in-process `{(abspath, st_mtime_ns, st_size): sha256}` cache across syncs; skip re-hashing when mtime+size are unchanged. Preserves content-hash dedup correctness (any real edit bumps mtime/size) while cutting a full-tree re-read down to just the changed files.

## 🟢 Low severity

### 3 & 4. Orphaned panels — `Search.tsx` and `Ingest.tsx`
**`web/src/Search.tsx:86` · `web/src/Ingest.tsx:69` · dead-code**

Both full panel components (default exports + their glyph/format helpers) are **never imported or mounted** — repo-wide grep for `import Search`/`import Ingest` returns nothing. `App.tsx`'s own comment says they "stay in the tree for U2/U3 to absorb but are not mounted here" — and that absorption is now **done**: `FindTurn.tsx` provides search-as-a-chat-turn and `Composer.tsx` provides ingest-as-a-chat-turn (reusing the same `sourceSnippet`/`ingestClient` helpers, which stay live). ~454 LOC of superseded scaffolding.

**Fix:** Delete both files. (`Ingest.tsx`'s `FILE_TINTS` map also duplicates `Library.tsx`'s `TINTS`, but it dies with the file.)

### 5. LLM completer opens a new `httpx.AsyncClient` per call
**`src/sift/adapters/llm/openai_compat.py:155` · network-round-trips**

`_post` wraps every request in `async with httpx.AsyncClient(...)`, so each call builds and tears down a pool. The completer is a long-lived per-container singleton, and `complete_with_tools` is called repeatedly against the *same* endpoint inside the answer tool-loop (`answer.py:800`, up to `answer_max_tool_calls`, plus a repair call at `:966`) — so every round-trip pays a fresh TCP+TLS handshake with no keep-alive reuse.

**Fix:** Hold one instance-level `AsyncClient` reused across `_post` calls (as the embedder already does within a batch). *Low severity:* the per-call client is documented as intentional, LLM latency dwarfs handshake cost, and localhost deployments often skip TLS — but the reuse is a free win under the tool-loop.

### 6. Up to 4 `stat` syscalls per file
**`agent/sync.py:208` · redundant-syscalls**

Per collected file: `_by_mtime` calls `os.path.exists` + `os.path.getmtime` (2 stats, over the *whole* candidate set incl. later-skipped files), then `_skip_oversized` calls `os.path.getsize` and `_modified_iso` calls `os.path.getmtime` again — up to **4 `stat()`s where one `os.stat()`** yields `st_mtime` + `st_size` together. Multiplies on large trees / network filesystems, and repeats every watch-mode sync.

**Fix:** `os.stat()` each file once (try/except for the vanished-file case) and thread `st_mtime`/`st_size` through the sort, oversize check, and ISO stamp. *Low:* each file is already fully streamed for SHA-256, which dominates.

### 7. Find-turn snippet regex re-runs per render / per streamed token
**`web/src/FindTurn.tsx:71` · web-rerender**

`FindTurnCard` and `FindRow` aren't memoized, and each `FindRow` recomputes `collapseWhitespace(hit.text)` + `highlightQueryTerms` (which builds a fresh unicode `RegExp` and splits the snippet) on every render. Since `Chat.tsx` re-renders the whole thread on **every streamed `answer_delta` token**, a thread containing an earlier Find turn rebuilds/splits the highlight regex for all its up-to-30 hits on **every token**. This is the *exact* class of issue round-1 fix #3 eliminated for `ChatMarkdown` via `React.memo` — the analogous Find/source renderer was left uncovered.

**Fix:** Wrap `FindTurnCard` in `React.memo` (the `turn` prop is referentially stable for untouched turns, so unchanged Find turns skip re-render during streaming); `useMemo` the per-hit snippet/highlight keyed on `hit.text + query`. Same treatment applies to `Chat.tsx`'s `SourcesPanel` snippet loop (needs `useCallback` on its toggles to benefit).
