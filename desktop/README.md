# Condense Desktop (Tauri 2 shell)

Standalone desktop launcher for Condense: a small Tauri 2 app that (in **local** mode) downloads
and supervises its own backend — the PyInstaller engine bundle, a `llama-server` sidecar serving
a local `bge-m3` GGUF embedder, and the folder-watching ingest agent — or (in **client** mode)
just talks to any Condense server the same way the browser build does (base URL + token). See
`docs/Quentin/active/machine.md` for the full design (D60–D64), the AppConfig schema, and the
Tauri command contract.

This directory is scaffold-only tonight (T4): the window, the app icon, the provisioning
manifest, and a Linux bundle that boots the existing `web/` UI. Rust-side provisioning +
process supervision (`config.rs` / `provisioning.rs` / `backend.rs` / `agent.rs`) land in T5.

## Layout
```
desktop/
  package.json              @tauri-apps/cli devDependency, `npm run tauri ...`
  Dockerfile.builder         builder image (webkit2gtk-dev + rust + node + QA tooling), no host install needed
  icon/                      brand mark SVG + composed 1024x1024 source PNG (`tauri icon` input)
  provisioning/manifest.json embedder/model/engine download manifest (schema 1, see machine.md)
  src-tauri/
    Cargo.toml, build.rs, tauri.conf.json
    src/{main,lib}.rs        thin main -> `condense_lib::run()`, builder registers shell+dialog plugins
    capabilities/default.json  shell spawn/kill scoped to the sift-agent-cli sidecar, dialog defaults
    icons/                   generated by `tauri icon` (gitignored inputs stay in ../icon/)
    binaries/                sift-agent-cli sidecar binary, per-target-triple (gitignored; see below)
```

## Prerequisites for a native (non-Docker) build
This host has neither `libwebkit2gtk-4.1-dev` nor sudo, so local builds happen in Docker (below).
On a machine that *does* have these installed natively, you only need:
- Rust stable (`rustup`), Node.js 22+, and the Tauri 2 Linux runtime deps: `libwebkit2gtk-4.1-dev`,
  `libgtk-3-dev`, `libayatana-appindicator3-dev`, `librsvg2-dev`, `libxdo-dev`,
  `libsoup-3.0-dev`, `libjavascriptcoregtk-4.1-dev`, `build-essential`, `pkg-config`, `libssl-dev`,
  `fakeroot`, `dpkg-dev` (for `.deb`).
- Then: `npm install && npm run tauri build` (or `npm run tauri dev` — brings up the Vite dev
  server per `beforeDevCommand`).

## Docker build (the way this host builds)
Build the builder image once (rebuild whenever `Dockerfile.builder` changes, ~4 min):
```bash
docker build -t condense-tauri-builder desktop/ -f desktop/Dockerfile.builder
```

Run a build against this worktree, with persistent cargo/npm caches so repeat builds are fast:
```bash
docker run --rm \
  --memory=5g --memory-swap=5g \
  -e CARGO_BUILD_JOBS=2 \
  -e CARGO_HOME=/work/.cargo-home \
  -e HOME=/work/.home \
  --user "$(id -u):$(id -g)" \
  -v "$(pwd)":/work \
  -v ~/.cache/condense-desktop-assets/builder/cargo-cache:/work/.cargo-home \
  -w /work/desktop \
  condense-tauri-builder \
  bash -lc "npm ci --prefix ../web && npm ci && npx tauri build"
```
Bundles land in `desktop/src-tauri/target/release/bundle/{deb,appimage}/`. `beforeBuildCommand`
in `tauri.conf.json` also runs `npm --prefix ../../web run build` itself, so a manually-built
`web/dist` isn't strictly required first — but building it once ahead of time (`npm ci && npm run
build` inside `web/`) makes failures easier to isolate.

`cargo check` / `cargo fmt --check` (same container, `-w /work/desktop/src-tauri`):
```bash
docker run --rm -e CARGO_HOME=/work/.cargo-home -e HOME=/work/.home \
  --user "$(id -u):$(id -g)" -v "$(pwd)":/work \
  -v ~/.cache/condense-desktop-assets/builder/cargo-cache:/work/.cargo-home \
  -w /work/desktop/src-tauri condense-tauri-builder \
  bash -lc "cargo fmt --check && cargo check"
```

### Icon pipeline
`desktop/icon/condense-mark.svg` is a static, standalone recreation of the animated brand mark
(`web/src/Logo.tsx`) — same geometry/colors, no CSS keyframes. `desktop/icon/icon-source.png` is
that mark composed onto a 1024x1024 rounded-square tile filled with the app's near-black
(`#0b0a10`, the same shade `Logo.tsx` uses for its input dots / `index.css` uses for `--text-h`),
mark centered at ~62% of tile width. Regenerate the platform icon set from it:
```bash
docker run --rm --user "$(id -u):$(id -g)" -v "$(pwd)":/work -e HOME=/work/.home \
  -w /work/desktop condense-tauri-builder \
  bash -lc "npx tauri icon icon/icon-source.png -o src-tauri/icons"
```

### Boot-smoke (screenshot the running app under Xvfb)
**`xvfb-run` and `dbus-run-session` both hang in this container** — with no init process, the
container's command runs as PID 1, and `xvfb-run`'s internal `wait` (blocking until Xvfb signals
readiness via `SIGUSR1`) never unblocks; `dbus-run-session` hangs the same way. The workaround:
start Xvfb ourselves with a plain `sleep` instead of the signal handshake, and skip D-Bus entirely
— the app tolerates the missing session bus (just a non-fatal AT-SPI warning on stderr). Save as
e.g. `/tmp/boot-smoke.sh` and bind-mount it in:
```bash
#!/bin/bash
export DISPLAY=:99
Xvfb :99 -screen 0 1280x800x24 -nolisten tcp &
XVFB_PID=$!
sleep 2
export WEBKIT_DISABLE_COMPOSITING_MODE=1
cd /work/desktop/src-tauri/target/release/bundle/appimage
APPIMAGE=$(ls Condense_*.AppImage)
./"$APPIMAGE" --appimage-extract-and-run &
APP_PID=$!
sleep 8
import -display "$DISPLAY" -window root /work/desktop/qa-boot.png
kill "$APP_PID" 2>/dev/null; sleep 1
pkill -9 -f "Condense_0.4.0" 2>/dev/null; pkill -9 -f AppRun 2>/dev/null
kill "$XVFB_PID" 2>/dev/null
```
```bash
docker run --rm --user "$(id -u):$(id -g)" -e HOME=/work/.home -v "$(pwd)":/work \
  -v /tmp/boot-smoke.sh:/tmp/boot-smoke.sh:ro -w /work condense-tauri-builder \
  bash /tmp/boot-smoke.sh
```
(`qa-boot.png` is gitignored — a QA artifact, not a committed asset.)

## CI
`.github/workflows/build-desktop.yml` (T3, `feat/desktop-ci`) matrix-builds the `desktop` job on
ubuntu-22.04/macos-latest/windows-latest via `tauri-action`, publishing `.deb`/`.AppImage`/`.dmg`/
`.msi`/`.nsis` artifacts (and release assets on `v*` tags). It builds the `sift-agent-cli` sidecar
per target-triple and drops it into `desktop/src-tauri/binaries/` before invoking Tauri, exactly
like the manual steps above.

## Provisioning manifest
`desktop/provisioning/manifest.json` (schema 1) is the config-driven source of the engine /
embedder / model download URLs the Rust shell fetches on first run (T5). Tonight's values for the
`llama-server` (b9878) + `bge-m3` GGUF embedder are real and sha256-verified for Linux x64; engine
URLs point at the `v0.4.0` release naming and stay `null` for sha256/size until that release
exists (see the manifest's own `_notes`). `AppConfig.manifest_url: null` resolves to this file's
raw GitHub URL on `main`; overriding it is how local E2E testing points the app at a locally built
bundle instead.
