How it works

One pipeline, four steps

Every file becomes searchable knowledge; every query returns a single, sourced answer.

01 · Ingest

Parse & chunk

Point the agent at a folder. PDFs, Office docs, Markdown and more are parsed to text and split into overlapping chunks.

02 · Embed

Vectorize into libSQL

Each chunk is embedded with bge-m3 and stored (vectors, metadata and dedup) in one libSQL database with native vector search.

03 · Rerank

Retrieve wide → top-1

A query pulls a wide candidate set, then a reranker distills it to the single best passage, not ten blue links.

04 · Recap

Answer + source

An optional LLM writes a short recap grounded in that passage, returned with its exact source path.

Why Condense

Minimal by design

Single best result

Retrieve wide, rerank, return one answer with its source, not a page of links.

Self-contained

libSQL native vectors hold data, metadata and dedup. No external vector database.

Ports & adapters

Every seam is an interface. Swap the embedder, store or reranker without touching callers.

Config-driven

One typed settings object is the single source of truth. Behaviour changes via config, not code.

Local-first

Your files, your machine. Inference runs over plain HTTP against endpoints you control.

Multilingual-ready

bge-m3 keeps German and French open, so you can add languages later without re-embedding.

Architecture

Clients talk to one API

The app is pure Python and ships no ML. Embeddings, rerank and the recap LLM are external HTTP endpoints you configure and swap; only libSQL travels with it.
